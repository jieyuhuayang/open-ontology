## 1. 入门概念

Open Ontology 平台中的数据，可以理解为分别存在于两个层面：数据层（data layer） 和 对象层（object layer）。

### 1.1 数据层

在数据层中，数据存储在 **数据集（datasets）** 里，可以表示各种类型的数据——包括结构化、非结构化与半结构化数据：
- 结构化（structured）（表格型）数据：由包含表格数据的文件组成，并包含关于数据集列的元数据。这些元数据会作为模式（schema）与数据集一同存储。
- 非结构化（unstructured）数据：由图片、视频或 PDF 等文件组成，但没有关联的 schema。
- 半结构化（semi-structured）数据：包含 XML 或 JSON 等文件格式。虽然可以为这些格式应用 schema，但出于性能与易用性考虑，更推荐在下游数据转换中推断为表格 schema。

数据集一般来源于组织内的数据源，并同步进入平台；当然，你也可以通过上传已获批准的数据或设想（notional）的数据来创建自己的数据集。

Open Ontology 平台中的每个数据集都会维护一份关于其如何生成的记录，从而确保数据的来源始终被保留且可追溯。这一概念称为 **数据血缘（data lineage）**。

* 平台会记录哪些 **输入数据集（input datasets）** 被用于生成哪些 **输出数据集（output datasets）**。这让你始终能够知道某条数据来自哪里，并理解数据是如何被使用的。

* 平台会追踪用于生成每个输出数据集的 **逻辑（logic）**。例如，你可能对某个输入数据集进行过滤以生成更小的输出数据集；该过滤逻辑会被保留并在平台中可见。

你可以通过平台的众多 **应用（applications）** 与数据交互。当你使用这些应用时，你产出的任何内容——无论是数据集、代码还是分析——都会作为平台中的一种 **资源（resource）** 被保存。资源会组织在 **项目（Projects）** 中；项目既用于把相关工作分组和组织起来，也作为权限边界。

### 1.2 对象层（本体 / Ontology）

在对象层（也称 Ontology）中，数据以 **对象（objects）** 和 **链接（links）** 的形式存储。对象表示现实世界中的概念，例如飞机、车辆或客户；链接表示对象之间的关系。对象层会把存储在表格型数据集中的数据（行与列）转换为一组组织内任何人都能理解的概念体系。



除了让数据更易理解，把数据从数据集转换为对象与链接，还能解锁一整套与对象交互的工具能力。你可以定义 **动作（actions）** 来描述组织内的人可以如何更改对象。这使你能够构建 **应用（applications）**：从对象中读取数据，并将用户的决策回写（capture）到系统中。



对象、链接与动作的定义共同构成所谓的 **本体（Ontology）**——你组织的数字化表示。



## 2. 本体相关概念



> **本体的定义，包括 2 类组成要素：**
>
> * 语义要素（semantic elements）：表示“世界中有什么”，包含对象、属性、链接。
>
> * 动态要素（kinetic elements）：表示““世界中发生什么、如何变化”，包含动作、函数、动态安全控制

### 2.1 **对象类型 (Object Type)**

1. **对象类型 (Object Type)**：对象类型是对现实世界真实实体或事件的抽象，对组织而言，是他们真正关心的重要“事物”。包含以下组成部分：

   1. 主键：唯一标识符

   2. 属性：对象类型的特征、状态或度量。每个 Object Type 可包含最多 2000 个 属性

   3. 显示元数据用于 UI 展示的名称、描述

   4. 状态标记：active（活跃）、experimental（实验性）、deprecated（已弃用）、

2. **对象（object）**：对象指某个对象类型的**单个实例，**&#x4E00;个对象对应现实世界中的一个具体实体或事件。

3. **对象集（object set）：**&#x6307;多个对象实例的集合；也就是说，对象集代表一组现实世界中的实体或事件。

4. **对象类型分组（Object type groups）：**&#x4E00;种用于分类的基础机制（classification primitive），帮助用户更好地在本体（Ontology）中进行搜索与探索。

> e.g.1 在 **Ontology Manager** 中，你可以创建一个 **Employee（员工）对象类型**，用来定义“所有员工”（即该类型的所有对象）的特征。**对象**则指 Employee 对象类型的某个单个实例，例如员工“Zhaoyayun”、“zhangxue”。像“所有已转正/已获得任期的员工（All tenured employees）”这样的对象分组，就代表一个**对象集**。

> e.g.2 同样地，在 Ontology Manager 中，你也可以创建一个 **Flight（航班）对象类型**，用来定义“所有航班”（即该类型所有对象）的特征。**对象**指 Flight 对象类型的某个单个实例，例如“JFK → SFO 2021-02-24”或“TLV → LHR 2020-04-16”。像“所有已到达航班（All arrived flights）”这样的对象分组，就代表一个**对象集**。



### 2.2 属性（Properties）

1. **属性：**&#x7528;于定义对象类型的**特征、状态或度量**。

   1. 属性本身不能独立存在，**必须依附于具体对象**。

   > 1) 例如，一个“员工（Employee）”对象可能具有如下属性：
   >
   > 2) 工号
   >
   > 3) 入职日期
   >
   > 4) 职位
   >
   > 5) 部门
   >
   > 6) 薪资

   * 不仅包含静态属性（String、Integer、DateTime、Timestamp、Array、Struct、Map、Vector、Geopoint 、TimeSeries），还必须支持**函数支持的属性 (Function-backed Properties)** ，可能不是数据库中的一列，而是通过实时调用 Python 函数或 ML 模型计算得出的动态值。

2. 属性值（property value）：某个对象上该属性的具体取值。在本体管理器中，属性值为对象类型添加底层数据源后，才会被创建并在用户应用中展示。

3. 支持的属性类型列表可参见《Supported property types》

4. 共享属性（shared property）：共享属性是一种可以在本体中多个对象类型上复用的属性，实现跨对象类型的一致数据建模，并集中管理属性元数据（property metadata）。

   > 例如，在 Ontology Manager 中，你可能同时有 **Employee&#x20;**&#x548C; **Contractor&#x20;**&#x4E24;个对象类型，它们都包含 **start date** 这一属性。通过创建一个 **start date 共享属性**并在两个对象类型中复用它，你可以用**一致的属性定义**来建模数据，并且只需在**一个位置**更新 start date 的**元数据（metadata）**，而无需分别在每个对象类型上单独维护。





### 2.3 链接类型（Link Type）

1. **链接类型（link type）：**&#x5B9A;义了对象之间的语义关系，从而构建一个反映现实世界结构的关系网络。

   > 典型链接类型例子包括：
   >
   > * 雇佣关系：员工 → 任职于 → 部门
   >
   > * 所有权关系：个人 → 拥有 → 资产
   >
   > * 包含关系：订单 → 包含 → 商品

   1. 注意：**不支持在不同 Ontology 之间的对象类型建立链接**。在这种情况下，通常应考虑使用一个**共享 Ontology（shared Ontology）**。

2. **链接（link）**&#x6307;的是在两个具体对象之间的一次单个链接类型的实例。

   1. 链接也可以存在于**同一对象类型的两个对象之间**。例如，可以在 `Employee` 对象类型自身之上定义一个 `Direct Report ↔ Manager` 的链接类型。

   2. 链接本身也可以拥有属性（例如，“关联”发生的时间、权重或原因）。

3. Ontology 所依赖的概念，在数据集结构中都有对应的类比概念：

   * 在 Ontology 中，**链接类型（link type）** 的定义，类似于在两个数据集之间定义一次 **join（连接）**；

   * 在 Ontology 中，**链接（link）** 的定义，则类似于在连接后的数据集中，一行数据与另一数据集中同一行字段的对应关系。

   > 例如，你可以将 `Employee` 数据集与 `Company` 数据集进行 join，以探索 `Employees` 与其 `Employers` 之间的关系。在 join 后的数据集中，将 “Melissa Chang” 与其雇主 “Acme, Inc.” 连接在一起的一行数据，就对应着 Ontology 中的一个链接实例。



### 2.4 动作类型（Action Type）

Action 是一次事务。

Action Type 定义了用户在一次操作中“可以对对象、属性值以及链接所执行的一组变更，以及执行动作时伴随的**副作用（Side Effect）行为**”。

Action Type 的“逻辑”是通过 **Rules（规则）** 来定义的。

Action Type 包括：

* **事务封装**：行动不仅仅是 `UPDATE` 语句。它是一个包含验证逻辑、副作用（Side Effects）和数据回写的原子单元。

* **回写 (Write-back)**：行动的核心职责是将变更同步回底层记录系统（System of Record, SoR），如 ERP 或 CRM，同时更新 Ontology 的缓存。

* **安全性与审计**：任何行动的执行都必须通过权限检查（“谁可以执行此操作？”），并且产生不可篡改的审计日志（Audit Log）。



### 2.5 接口（Interfaces  ）

**接口（Interface）** 是一种本体（Ontology）类型，用于描述某个**对象类型（Object Type）的形状（shape）及其能力（capabilities）**。

接口提供对象类型的**多态性（polymorphism）**，使得对具有共同形状的对象类型能够进行一致的建模与交互。

### 2.7 安全与权限相关概念

#### 2.7.1 组织（organizations） 与空间（spaces）

##### 2.7.1.1 组织

**组织是一种强制性的访问控制措施**，一个平台实例里由一个或多个组织构成，组织把空间、本体、项目、用户等资源严格隔离，只有满足这些组织访问要求的用户才能访问。

* **用户归属规则**：每个用户只属于一个主组织，但可以作为访客（guest）加入多个组织。

* **继承方式**：Organizations 会通过**文件层级与直接依赖关系**继承。

* **与 Markings 的差异**：

  * 组织保护的信息范围包括：空间、本体（ontologies）、项目、用户、用户组、标签类别（tag categories）和集合（collections）。但是，**单个资源无法绑定到某个组织**。相比之下，标记只能应用到项目与资源。

  * 组织保护的信息会遵循**跨组织可发现性（cross-organization discoverability）**&#x89C4;则。平台管理员可以允许或禁止用户查看本组织之外其他组织的名称、用户与用户组。

  * 用户必须隶属**唯一一个**组织；而标记不存在“用户必须拥有/隶属某个标记”的要求。

##### 2.7.1.2 空间

* **本质：Space 是项目（Projects）的高层容器**，用于承载一组 Projects，并且**共享同一个 Ontology**，为某一类工作目的（同一组组织共享）提供统一工作区。

* 空间与组织的关系：空间有一个“可访问的组织列表/访问要求（access requirements）”。用户要访问这个空间，必须至少满足其中某个组织要求；同时，空间也限定了空间内项目能使用的组织范围。

* 空间与本体的关系：1:1 映射。创建新 Space 时，会同时创建一个**同名 Ontology**，并且带有与该 Space 相同的 **organization markings**。

  * Private space → private ontology；Shared space → shared ontology。shared ontology 会继承 shared space 的**组织 markings**与**role grants**。

> 一个直观例子：
>
> * 你在 **Org A** 内建一个 **private space “Trading”** → 系统自动建一个 **private ontology “Trading”**，只有 Org A（及其允许的 guest）能访问。
>
> * 你要和 **Org B** 协作 → 建一个 **shared space “Joint-Workflow”**（受 Org A + Org B 限制）→ 自动生成 **shared ontology “Joint-Workflow”**，两边都能在同一套本体上协作。

* 常见模式：多数组织只需要一个 space，space 内再用 markings 和 roles 进一步细分 project 访问。

#### 2.7.2 项目（Projects） 与角色（roles）

##### 2.7.2.1 项目与资源（Projects and resources）

项目（Projects）可以理解为共享工作的“bucket”，由于项目边界会强制执行安全控制，项目是组织数据、并在安全空间内实现开放协作的关键手段。

每个项目都是一个协作空间，用于围绕特定目的组织用户、文件与文件夹。

项目的设计应使在同一项目内协作的用户对内容拥有大体一致的访问范围，但对内容具备不同层级的权限——例如，你的项目里所有人默认都是 Viewer（查看者）角色，但只有某个特定用户组在所有项目资源上拥有 Editor（编辑者）角色。

项目会强制安全边界，使工作能够被“收敛”在一个空间内。工作（转换、分析等）在项目中完成，工作产物与其逻辑会一同存放在该项目中。

![](<images/0 概念定义-image.png>)

由于工作与其输出必须位于同一项目中，如需复用或基于其他项目的工作继续构建，你必须使用引用（References）。

为简化权限管理，我们建议在项目级别向用户组（group）授予角色。通过用户组管理项目权限，可以将拥有相同权限的一组用户统一管理，减少逐个用户授予角色带来的杂乱。项目内容会继承项目级别授予的角色（roles），从而为用户提供对项目内容较为一致的访问能力。







##### 2.7.2.2 引用（References）

要构建一个有用的流水线，你往往需要使用来自其他项目的数据集。此时，建议通过**文件引用（file references）**&#x6765;使用其他项目的数据集。

![](<images/0 概念定义-image-1.png>)

添加文件引用后，只要你拥有该用例所需的角色，就可以在本项目中使用上游数据集。一旦导入，你的同事无需访问上游项目，也能看到在你项目中派生的数据集——前提是他们满足该数据集上的任何组织（Organizations）与标记（Markings）要求。

![](<images/0 概念定义-image-2.png>)

在上图中，我们在 `Flight Delays [Transform]` 项目中引用了来自 `Flight Control System [Datasource]` 项目的 `flights` 数据集。如果我们将某个用户加入 Transform 项目，该用户将能够查看 `delays` 数据集，并在其之上构建更多 transform。可是，如果要查看原始 `flights` 数据集，或在其之上构建额外的转换，则仍需要在上游项目上拥有 `Viewer` 角色。

项目与引用有助于组织协作：项目所有者可以轻松地将用户加入自己的项目，并确保他们拥有适当权限。

通常，要向某项目添加文件引用，你需要在源项目上拥有 `Viewer` 角色，并在目标项目上拥有 `Editor` 角色。

##### 2.7.2.3 角色（roles）

角色是一组权限集合，用于授予对资源的不同级别访问能力。角色属于**自主权限（discretionary permission）**，通常在项目级别授予，以便在项目范围内的所有资源上提供一致的能力。

然而，强制控制——组织（Organizations）与标记（Markings）——将**始终**阻止不符合条件的用户访问资源，无论其角色为何。

默认情况下，角色从高到低依次为：Owner（所有者）、Editor（编辑者）、Viewer（查看者）和 Discoverer（发现者）。每个角色都可以为其他用户分配**相同或更低**的角色。

> 例如，Owner 可以为其他用户授予 Owner、Editor、Viewer 或 Discoverer；而 Discoverer 只能为其他用户授予 Discoverer。默认角色可被定制，包括新增完全自定义角色（new roles）。

同样重要的是，与强制控制一样，角色授予会继承到子资源。例如，在项目或文件夹上授予某用户 Viewer 角色，会让其对该项目或文件夹内包含的所有资源都拥有 Viewer 权限。通常会以用户组为单位在项目上授予角色。

![](<images/0 概念定义-image-3.png>)

#### 2.7.3 用户和用户组（Users and groups）

* **用户（user）**：指已通过身份验证、并可访问本平台的个人。用户通常由外部身份提供方定义。用户拥有属性（attributes），即关于该用户的公开、结构化信息。

* **用户组（group）**：一组用户的集合，让安全组管理更容易，对 Project（项目）与资源的访问权限通常授予组而非单个用户。组通常由外部身份提供方定义，并在整个平台中统一应用。

#### 2.7.4 标记（Markings）

标记（Markings）为平台中的文件、文件夹和 Project 提供额外一层访问控制。标记会定义一组“资格条件”，将资源的可见性与可执行操作限制在满足这些条件的用户范围内。

要访问某个资源，用户必须同时属于施加在该资源上的全部标记，才能获得访问权限。平台管理员通常会在某个\*\*组织（Organization）\*\*内管理标记。

对某个标记的访问是二元的（全有或全无）。无论用户具备何种角色（role），只要用户不满足所有标记要求，就无法以任何方式访问该文件。标记旨在让数据保护官能够集中管理并审计“究竟谁可以访问某一类数据”。



> 标记的一个常见用例是限制对\*\*个人身份信息（PII）\*\*的访问。例如，你可能规定一组用户只有在完成一系列培训后才有资格访问敏感的 PII 数据。平台管理员可以创建一个 `PII` 标记并将其应用到敏感数据集上。这样就能确保对 PII 的访问仅限于合规用户，且无法被共享到该群体之外。



标记属于**强制控制（mandatory control）**，而角色属于**自主控制（discretionary control）**。强制控制通过要求用户必须拥有某个特定标记才能访问数据，从而**限制**访问。要移除某个标记，需要该标记自身的 **Expand Access（扩展访问）权限——这是一项集中管理的权限。**

> **例如，即便某用户在一个标有 `PII` 标记的数据集上拥有 Owner 角色，仅凭 Owner 角色也不能**移除该标记；除非该用户同时拥有该标记的 Expand Access 权限。相反，自主控制用于**扩展**访问，通常通过数据共享工作流授予，缺少集中式限制。例如，任何在资源上拥有 Owner 角色的用户，都可以把 Owner 角色授予其他用户或群组。

由于标记是“合取”的（布尔 `AND`），用户必须同时属于某文件、文件夹或 Project 上的**所有**标记，才能获得访问权限。
